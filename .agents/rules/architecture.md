# 아키텍처 규칙 — campus-hackathon-2026

> 스택·아키텍처 미확정 단계. 주제와 스택이 정해지면 본 문서에 레이어·데이터 흐름·계약을 박제하고, 중요한 결정은 `docs/architecture/decisions/`에 ADR로 근거를 남김.

## 결정 전 채울 항목

- **레이어**: 화면 / API / 도메인 로직 / 데이터 경계를 어떻게 가르는가.
- **데이터 소스**: 실제 DB인가 mock/정적 fixture인가(해커톤 데모는 mock으로 충분한 경우가 많음).
- **API 계약**: 엔드포인트·요청/응답 타입·에러코드. 화면이 이 계약으로 분기하도록 안정 설계.

## 스코프 제약(오버엔지니어링 금지 — ponytail)

해커톤 데모 기준. 시간이 가장 큰 제약이므로 아래는 사용자/팀 합의 없이는 도입 금지:

- 실제 DB·ORM(데모에 mock으로 충분하면 mock 우선)
- 계정·로그인·세션·결제(데모 범위 밖이면 도입 금지)
- 상태관리 라이브러리(내장 상태·서버 컴포넌트로 충분)
- 단일 구현 인터페이스·팩토리·불필요 config 추상화
- 실시간 소켓·서버측 페이지네이션 등 "있으면 좋은" 기능

데모에서 실제로 보여줄 핵심 차별 요소 하나에 시간을 몰고, 나머지는 최소 구현으로 둠.
